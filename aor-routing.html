AOR — Advance Omnichannel Routing
AOR is the core routing layer for inContact/CXone. It handles voice ACD and digital (IFM) contacts through a shared matching pipeline called FindMatch. AOR is the default routing path for all tenants. EAIR is one of several agent selection methods (alongside Least Occupied, Most Occupied, Longest Available) configured per-skill within the Agent Selector service.
Key Concepts
Contact Routable (CR)
Event telling FindMatch a contact is ready to be matched. Published by VC (voice) or UR Lambda (digital).
Agent Routable (AR)
Event telling FindMatch an agent is available. VC publishes these as agent state changes.
matchrequests stream
{env}-findmatch-matchrequests Kinesis — single intake point for ALL CR and AR events (voice + digital).
{cluster}-Matches SQS
Output queue for accepted voice + digital matches. VC consumes these to assign agent to contact.
Contact Skill Routable
Outbound dialer message telling the matching system how many agents are needed for outbound calls on a given skill. Contains contactCount, matchMode (ON/OFF/UseCount), priority, and queueId. Not a per-contact signal.
Voice / ACD Path
Traditional inbound call routing
Inbound call arrives → VC evaluates skill requirements → VC publishes Contact Routable to matchrequests stream → FindMatch makes match → result published to {cluster}-Matches SQS → VC joins media legs
Digital (IFM) Path
Email, Chat, SMS via DFO platform
DFO contact → ContactSync → UR Lambda converts to contact_routable_v6 → same matchrequests stream → FindMatch matches → {env}-digital-matches → UR Lambda → VC accepts/rejects
Legacy: SQL FindMatch (SQLFM)
Static Delivery — being migrated off
Original FindMatch lived as a SQL stored procedure inside the VC monolith database. Tenants still on SQLFM use "Static Delivery" (single channel). Active migration to AFM is in progress.
FindMatch Engines — Then → Now → Future
LEGACY
SQLFM
SQL stored procedure in VC monolith database. Single-channel "Static Delivery". No horizontal scaling. Being EOL'd — customers must self-migrate to Dynamic Delivery.
MIGRATING OUT
VFM — Voice FindMatch
ECS service on EC2. Shard-partitioned (1 Kinesis shard → 1 node; many tenants share a node). Manually scaled by Kinesis shard count. In-memory state. Hot-shard problem. 3–4 day release cycles.
CURRENT / TARGET
AFM — Advanced FindMatch
Microservices on shared EKS. Kafka-based internal comms. Load-based autoscaling. Cell architecture. Handles both inbound and outbound. Replaces VFM and SQLFM.
AOR Retry Service
Subscribes to {env}-findmatch-matchrequests. A Contact Routable adds the contact; a Not Routable removes it. Retries with exponential backoff (doubles each time until max). After max is reached, retries at max interval indefinitely. Sets retry=true on re-published events.
AOR vs EAIR — Key Distinction
AOR (this diagram) = default path for ALL tenants. Skill-based deterministic matching via FindMatch.

EAIR is one of several agent selection methods configured per-skill in the Agent Selector service (alongside Least Occupied, Most Occupied, and Longest Available). It is not a separate system — it is integrated at the Agent Selector step. A customer enables EAIR per-skill; other skills can use a different method simultaneously.
Voice ACD Routing — the original FindMatch flow. Skill-based, deterministic, driven by Contact Routable events. The VC owns the contact lifecycle; FindMatch is the matching engine. Click each step for details.

Agent Routables (AR) — the other half of matching
FindMatch needs both a Contact Routable (contact waiting) and an Agent Routable (agent available) to make a match. The VC continuously publishes Agent Routables to {env}-findmatch-matchrequests as agents change state (log in, go available, go busy). VFM uses these as an additional trigger to retry matching when no contact-side retry has fired.
IFM (Integrated FindMatch) — routes digital contacts through the same FindMatch engine as voice. The difference from standard Unified Routing: matching is deterministic (FindMatch) instead of pre-assignment (DFO engine). Retry logic is contact-driven (Contact Routable events) rather than agent-driven.
Standard Unified Routing (non-IFM)
DFO platform performs "pre-assignment" matching itself. Fire-and-forget sends candidates to VC. Retry is agent-routable driven (VC sends heartbeats). No AOR FindMatch involvement.
IFM — Integrated FindMatch
AOR's FindMatch engine makes the routing decision. Deterministic matching. Retry is contact-routable driven (AOR Retry Service). Needs integrated-findmatch-all feature set enabled on the tenant.
There are three generations of FindMatch engine, reflecting the AOR team's migration from monolith → ECS → EKS microservices. All three share the same Kinesis intake stream and SQS output queue interface, allowing gradual tenant migration.

② VFM — Voice FindMatch / Virtual FindMatch
ECS · EC2 · Shard-partitioned · Being migrated to AFM
Architecture
ECS service running on EC2 instances
Shard-partitioned: 1 Kinesis shard → 1 VFM node (many tenants share a node)
A tenant's events always land on the same shard/node (consistent hashing)
In-memory state per shard (routable contacts + agents for all tenants on that shard)
Scaled manually: 1 ECS task per Kinesis shard
Hot-shard problem: a large tenant can overload its shard's node
3–4 day release cycle (long due to ECS constraints)

③ AFM — Advanced FindMatch
EKS · Kafka-internal · Cell-based · Target state
Architecture
Microservices running in shared EKS (namespace: apps-routing)
Kafka-based internal event bus between services
Load-based horizontal autoscaling (Karpenter)
Cell architecture: each cell owns a shard of tenants

Continuous deployment (no multi-day release windows)
